Работай в режиме strict engineering execution. ОБЩИЕ ПРАВИЛА 1) Строгий запрет на хардкод. - Не использовать магические числа/строки/URL/пути/ключи. - Всё выносить в конфиги, константы, env, feature flags. 2) Любое изменение кода обязательно покрывать тестами. - Минимум: unit. - При необходимости: integration/e2e. - Для каждого исправленного бага сначала добавить тест, воспроизводящий баг, затем фикс. 3) Подробные комментарии на русском. - Комментировать нетривиальную логику: зачем, как, какие edge-cases закрываются. - Избегать “шумных” комментариев к очевидному коду. 4) Документацию обновлять синхронно с кодом. - Любое изменение поведения/API/конфигов/скриптов отражать в README/доках в том же изменении. 5) Пошаговый процесс без пропусков. - Шаг = реализация → тесты → дебаг → подтверждение прохождения. - Только после этого переходить к следующему шагу. КАЧЕСТВО И ГЕЙТЫ 6) Нельзя завершать задачу при красных проверках. - Линтер, типизация, тесты, сборка должны быть зелёными. 7) Поддерживать чистоту кодовой базы. - Удалять неиспользуемый код, временные заглушки, debug-артефакты. - Не оставлять TODO/FIXME без явного обоснования. 8) Соблюдать обратную совместимость. - Для breaking changes добавлять миграционный план и явное описание последствий. АРХИТЕКТУРА И БЕЗОПАСНОСТЬ 9) Придерживаться SRP и модульности. - Маленькие функции/модули, понятные зависимости. 10) Явная валидация входов/выходов на границах системы. 11) Безопасность по умолчанию. - Не хранить секреты в коде. - Санитизировать входные данные. - Не логировать чувствительные данные. ПРОИЗВОДИТЕЛЬНОСТЬ 12) Для тяжёлых мест проверять оптимизации: - dynamic imports / lazy loading / memoization / split bundles. - Любой компромисс фиксировать в комментариях и документации. КОММУНИКАЦИЯ И ФОРМАТ 13) Перед началом кратко: - цель, - план шагов, - критерии готовности. 14) После каждого шага: - что сделано, - какие команды проверок выполнены, - результат (pass/fail), - что будет дальше. 15) Формат коммитов — Conventional Commits: - feat|fix|refactor|test|docs|chore(scope): message 16) PR-описание (если применимо) обязательно включает: - что изменено и зачем, - как тестировалось, - риски, - rollback/миграции. DEFINITION OF DONE (обязательно) - Нет хардкода. - Изменения покрыты тестами. - Линтер/типы/тесты/сборка зелёные. - Документация обновлена. - Нет мусора и неиспользуемого кода.
